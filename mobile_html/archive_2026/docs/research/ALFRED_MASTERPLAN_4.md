### 5.2 Autonomous Decision Loop

Alfred runs a continuous background loop (every 60 seconds) that:

1. **Perceives** — checks system health, pending tasks, new events, sensor data
2. **Reasons** — evaluates goals, priorities, resource availability
3. **Decides** — selects the highest-impact action to take next
4. **Acts** — delegates to the appropriate Director agent
5. **Reflects** — logs outcome, updates success rates, adjusts strategy

```
┌─────────────────────────────────────────────────┐
│          ALFRED AUTONOMOUS DECISION LOOP         │
│                                                   │
│    ┌───────────┐                                  │
│    │ PERCEIVE  │◀── System health, events,        │
│    │           │    sensor data, market feeds      │
│    └─────┬─────┘                                  │
│          ▼                                        │
│    ┌───────────┐                                  │
│    │  REASON   │◀── Goals, priorities, history,   │
│    │           │    resource constraints            │
│    └─────┬─────┘                                  │
│          ▼                                        │
│    ┌───────────┐                                  │
│    │  DECIDE   │──▶ Select highest-impact action  │
│    │           │    With safety constraints         │
│    └─────┬─────┘                                  │
│          ▼                                        │
│    ┌───────────┐                                  │
```

### 9.1 The Tool Genesis Pipeline

Alfred's most powerful capability: **creating new tools autonomously**.

```
TOOL GENESIS FLOW:
──────────────────────────────────────────────────
1. IDENTIFY NEED
   Alfred detects a gap (user asked for something no tool handles)
   OR a feed item suggests a new capability worth building

2. DESIGN
   Genesis agent drafts tool specification:
   - Name (canonical ID, display name)
   - Parameters (input schema)
   - Implementation approach
   - Safety constraints

3. GENERATE
   Forge agent writes the code:
   - PHP function for VAPI tools
   - JavaScript handler for MCP server
   - Python wrapper for SDK

4. TEST
   Tester agent validates:
   - Unit tests pass
   - No security vulnerabilities (Sentinel review)
   - Performance benchmarks acceptable

5. REGISTER
   Genesis agent registers the tool:
   - Add to VAPI switch statement
   - Add to MCP server tools.js
   - Add to tool registry
   - Generate documentation

6. DEPLOY
   Deployer agent ships it:
   - Git commit with descriptive message
   - PM2 graceful reload for MCP server
   - Tool available within 60 seconds

7. MONITOR
   Oracle tracks usage:
   - Error rate, latency, adoption
   - If error rate > 10%, auto-rollback
──────────────────────────────────────────────────
```

## 16. SDK ARCHITECTURE DECISION

### The Question: One Unified SDK or Keep Separate?

**Answer: KEEP THEM SEPARATE.** The architecture is already correct.

```
┌─────────────────────────────────────────────────────────────────────┐
│                 ALFRED SDK ARCHITECTURE (FINAL)                      │
│                                                                      │
│   LAYER 1: REAL-TIME CONTROL (Browser)                               │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │  Game Engine SDK v3.0 (gositeme-game-engine.js)             │   │
│   │  22 modules — Three.js, WebXR, WebSocket, WebRTC            │   │
│   │  6 robotics modules — direct rosbridge ws://robot:9090      │   │
│   │  Runs entirely in browser. Zero server dependency.           │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   LAYER 2: MANAGEMENT PLANE (Server-side)                            │
│   ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐      │
│   │ Node.js SDK     │ │ Python SDK      │ │ PHP SDK         │      │
│   │ @alfredai/sdk   │ │ alfred-ai-sdk   │ │ alfredai/sdk    │      │
│   │ v2.0.0 — TS     │ │ v2.0.0          │ │ v2.0.0          │      │
│   │ 8 resources     │ │ 8 resources     │ │ 8 resources     │      │
│   │ REST → API v1   │ │ REST → API v1   │ │ REST → API v1   │      │
│   └─────────────────┘ └─────────────────┘ └─────────────────┘      │
│                                                                      │
│   All 4 SDKs include robotics. They serve DIFFERENT purposes:        │
│   • Game Engine = real-time 60fps robot control + 3D visualization   │
│   • API SDKs = robot fleet management, goal dispatch, sensor reads   │
│   Merging them would force every API user to pull in Three.js.       │
└─────────────────────────────────────────────────────────────────────┘
```

**Why separate is better:**
1. Game Engine is browser-only (2,525 lines, Three.js dependency). API SDKs are server-only (cURL/fetch).
2. They use different transports: WebSocket (rosbridge) vs REST API.
3. Developers use them differently: game devs want the engine, backend devs want the API client.
4. No code is shared between them — zero benefit from merging.
